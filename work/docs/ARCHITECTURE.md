# 架构说明

本作品不包含额外的大模型 API 客户端。比赛平台提供的任意通用 AI Agent
都是智能决策核心，只要它能够读取 Markdown 和文件、运行本地命令并写入
产物：

```text
平台通用 AI Agent
  -> INSTRUCTION.md
  -> work/skill/SKILL.md
  -> 只读分析 code/
  -> 临时隔离答案材料并完成盲分析
  -> 识别系统类型并生成风险加权覆盖计划
  -> 建立攻击面矩阵
  -> 单次运行采用多种独立策略
  -> 强制审阅并验证全部项目线索
  -> 探测运行时、命令、依赖与服务要求
  -> 隔离构建/测试或启动本地服务并执行健康探针
  -> 完整启动或分层降级验证
  -> 动态生成并执行测试
  -> 原生运行时覆盖证明关键产品路径执行
  -> 隔离副本中的定向变异测试验证断言强度
  -> 汇总独立确认与项目线索确认
  -> result/ + logs/
```

`openai.yaml` 是 Skill 生态要求的界面元数据，不表示作品依赖某个模型或
服务。核心指令均为普通 Markdown，辅助工具均为 Python 标准库脚本。

## 单次运行模型

五次评分运行互不累计。每次运行：

1. 生成独立 `run_id`。
2. 清理上一轮生成产物。
3. 从当前 `code/` 重新发现项目。
4. 在接触答案材料前固定首批候选和测试计划。
5. 对 15 类漏洞方向逐项判定适用性和优先级。
6. 审阅全部教程、CVE、解答和漏洞说明，逐条验证可操作线索。
7. 至少使用三种相关漏洞策略，复杂项目以四种为目标。
8. 对写入型命令使用隔离副本，对服务使用本地健康检查和动态探针。
9. 回补高风险覆盖缺口，完整运行目标为至少 95% 风险加权覆盖。
10. 使用可用原生工具测量目标代码覆盖，并对关键决策执行隔离定向变异。
11. 独立生成、执行、复测和报告。

## 复杂项目与降级

大型项目先按清单、子项目、入口和信任边界分层，再从输入、身份、解析、
授权、状态、数据汇、输出和外部组件建立攻击面矩阵。完整系统启动失败时，
依次尝试现有测试、子项目、公共调用面、隔离假实现和纯逻辑测试。一个模块
失败不会阻止其他模块或策略继续。

`runtime_runner.py` 将可能写入源码树的命令放入临时副本，使用缩减环境、
无交互输入、进程组超时清理和有界脱敏日志。服务模式只检查本地地址，运行
本轮生成的探针后自动关闭。所有尝试进入清单并由最终校验器核对。

候选测试可组合边界值、差分、变形、权限矩阵、状态机、故障注入、编码和
配置契约等方法。隔离假实现只能替代外部依赖，不能替代被验证的目标逻辑。
覆盖率证明测试到达真实产品路径；定向变异通过条件取反、边界变化、权限
判断移除和状态变化等方式检验断言是否真正约束关键行为。变异只发生在临时
副本，完整运行要求无未解决关键变异且分数至少 80%。

系统采用双轨模型：独立探索衡量未知漏洞发现能力，项目线索验证保证已知
风险不漏报。两类结果均按相同执行证据标准确认并计入 Bug 总数，只在来源
统计中分开。

95% 指标只表示已声明风险计划的动态测试覆盖率。未知系统不存在可直接证明
的“全部 Bug”分母，因此系统禁止把该指标宣传为 95% Bug 发现率。

## 数据边界

- 输入：`code/`
- 作品规则和工具：`work/`
- 生成产物：`result/`
- 审计记录：`logs/`

测试不得写回 `code/`。运行前后使用内容快照验证源文件未变化。
